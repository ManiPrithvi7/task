EMQX Cloud Serverless — X.509 and TLS notes (statsmqtt)
========================================================

NOTE (2026-04): EMQX-downloaded PEMs (e.g. emqxcloud-download-ISRG-Root-X1.crt) and DigiCert broker-ca
bundles were removed from data/ca/. The repo now uses a single regenerated StatsMQTT Root CA in
data/ca/ (see broker/README.txt for NanoMQ). If you still use EMQX Cloud, download the current CA/chain
from the EMQX console and set MQTT_TLS_CA_PATH to that PEM (DigiCert or ISRG as appropriate); do not
commit those files if they duplicate public roots — merge with tlsBrokerCa behavior as needed.

1) VERIFYING THE BROKER (your Node app / firmware trusts the SERVER)
--------------------------------------------------------------------
Your deployment host: *.ala.asia-southeast1.emqxsl.com
TLS chain uses DigiCert (e.g. "Encryption Everywhere DV TLS CA - G2" → DigiCert roots).

• For EMQX Cloud: use a PEM that verifies your broker hostname (DigiCert Global Root G2 often works).
• The app merges this PEM with Node’s Mozilla CA store for TLS (see src/utils/tlsBrokerCa.ts) so EMQX’s
  DigiCert chain does not fail with “unable to get local issuer certificate” when only one root PEM is set.

• The file from EMQX Downloads named emqxcloud-ca.crt is often ISRG Root X1 (Let’s Encrypt root).
  For some Serverless hostnames it does NOT complete the chain (verify return code 20) — use the chain
  EMQX documents for your exact endpoint.

Env (optional override):
  MQTT_TLS_CA_PATH=./path/to/your-emqx-trust-bundle.crt


2) X.509 CLIENT AUTH (broker trusts DEVICE certificates)
--------------------------------------------------------
In EMQX Cloud Console (Serverless supports X.509 per auth matrix):

  Access Control → Authentication → Add / configure “X.509 Certificate” (wording may be “Certificate”).

  Upload the PEM of the CA that SIGNS your device certificates (public CA cert only), e.g.:
    - Your StatsMQTT provisioning root: data/ca/root-ca.crt (from CAService), OR
    - The same PEM you distribute to devices as “device issuer CA”.

  Configure how EMQX maps identity (often CN → MQTT username / clientid). Match your firmware.

  If the console also offers TLS “two-way” / client CA for the listener, upload the SAME device-issuing CA
  there if required by the product wizard.

  Password auth: Serverless may still have the built-in user (e.g. proof / proof_dev). For cert-only devices,
  disable or reorder authenticators so X.509 is sufficient, per EMQX docs for your plan.


3) YOUR APP (.env)
------------------
  MQTT_BROKER=dfc176cf.ala.asia-southeast1.emqxsl.com
  MQTT_PORT=8883
  MQTT_USERNAME / MQTT_PASSWORD   ← often still required at MQTT layer until broker is cert-only

  If using client cert from app to EMQX: place PEMs and set paths or MQTT_TLS_*_BASE64 in Render secrets.


4) TROUBLESHOOTING CONNACK AFTER TLS OK
---------------------------------------
If logs show “<- packet cmd=connack” then “Bad username or password” (MQTT reason code 4) or
“Not authorized” (code 5), TLS and the broker hostname are fine — EMQX is rejecting the MQTT layer.

  • Code 4: CONNECT username/password do not match a configured authenticator (built-in DB, etc.).
    Fix: use the same MQTT_USERNAME / MQTT_PASSWORD as in the EMQX console (and statsmqtt .env).
    Do not send the cert CN as MQTT username unless the broker is explicitly configured for that
    (node-mqtt-client: leave MQTT_PEER_CN_AS_MQTT_USERNAME unset or 0 unless you know X.509 maps CN).

  • Code 5: often ACL or auth chain order (e.g. password check runs before cert). Reorder/disable
    redundant authenticators in EMQX so X.509 matches your intent.

  • getaddrinfo EAI_AGAIN: DNS flake; retry or fix network/VPN. Not a cert problem.


5) FIRMWARE
-----------
  • Trust server: same PEM you use for MQTT_TLS_CA_PATH (DigiCert chain for EMQX Cloud, or your Root CA for NanoMQ).
  • Present device cert + key signed by the CA you uploaded in step 2.
  • Use broker host + 8883; follow EMQX rules for username (CN vs fixed user).
